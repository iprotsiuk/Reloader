# Curated MainTown NPC Appearance Design

**Date:** 2026-03-09
**Status:** Approved for implementation

## Goal

Make the approved STYLE clothing/body/hair module set the default appearance source for `MainTown` civilians so the town population is procedurally varied without using broken or unreviewed combinations.

## Chosen Approach

Treat the reviewed STYLE pack as an approved module pool, not as a list of approved finished characters.

`MainTown` civilians should be generated by combining approved modules through explicit compatibility rules:

- body and presentation selection remains compatible with the current save record
- hair and beard choices come from approved gender/body-safe pools
- tops, bottoms, and outerwear come only from reviewed module ids
- layering rules prevent invalid combinations such as incompatible underlayers or missing required base layers
- banned or broken modules stay out of the runtime pool even if they still exist in the imported asset pack

## Why This Approach

- It preserves procedural variety instead of cloning demo-scene NPCs into the game.
- It turns the manual review pass into a durable runtime contract.
- It keeps save data stable because generated civilians still serialize resolved module ids rather than scene references.
- It gives us a clean path to hand-author special roles later without changing the civilian generator again.

## Approved Module Pool Contract

The canonical approved pool for this slice is the reviewed STYLE module subset already validated in the crowd-review work.

Runtime generation should treat that pool as:

- approved body/presentation options
- approved hair and beard options
- approved tops
- approved bottoms
- approved underlayer and outerwear rules
- approved material/color variant families
- banned module ids and banned pairings
- male bodies use male module families where the STYLE pack distinguishes them
- female bodies use female module families where the STYLE pack distinguishes them

This slice does **not** make named review-scene NPCs canonical runtime content.

The review scene remains upstream QA only:

- it proves which modules are safe
- it proves which combinations are safe
- it does not define production civilian identities

## MainTown Role Policy

### Random population

- `MainTown` general civilians should use seeded random combinations from the approved module pool.
- Different civilians may share modules, but generation should stay deterministic per seed/save.
- Pool-specific differentiation such as `cops` or `hobos` can stay shallow for now; the immediate goal is safe civilian variety.

### Authored roles

- Vendors and other authored scene roles will eventually become hand-authored fixed identities.
- For this pass, authored roles may still use fixed generated-looking combinations assembled from the same approved module pool.
- No authored role should depend on one named demo-scene test character.

## Runtime Contract

The current `CivilianPopulationRecord` fields remain the persistence contract for this slice:

- `baseBodyId`
- `presentationType`
- `hairId`
- `hairColorId`
- `beardId`
- `outfitTopId`
- `outfitBottomId`
- `outerwearId`
- `materialColorIds`
- `generatedDescriptionTags`

The change is in how those fields are chosen:

- old behavior: pick each field independently from flat arrays
- new behavior: pick from approved module pools with compatibility guards so the resolved record is always a valid assembled look

## Visual Application Contract

`MainTown` runtime civilians must visibly consume the generated appearance ids rather than carrying them as save-only metadata.

This slice should add a runtime appearance application seam that:

- reads the generated record ids
- enables the matching reviewed STYLE body/hair/clothing modules
- applies approved external material families
- enforces the same normalization and layering guards used by the review builder

## Validation Targets

- Generated civilians use only approved module ids.
- Generated civilians do not resolve banned pairings.
- Save/load preserves the same generated appearance ids.
- Runtime-spawned `MainTown` civilians visibly reflect the generated ids.
- Authored vendor and service roles are no longer backed by the old generic NPC visual source.

## Deferred

- police-, EMS-, ranger-, and worker-specific curated sub-pools
- hand-authored fixed vendor identities
- richer role/archetype weighting
- portraits, contract-intel appearance phrasing, and deeper descriptive tags
